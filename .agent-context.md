# Agent Context: ISPPOS

Concise entry point for AI agents working in this repository.

## Project Identity
- ISP POS / card-sales management app.
- Expo React Native app, JavaScript only, not TypeScript.
- Entry flow: `App.js` -> providers -> `src/navigation/AppNavigator.js` -> screens -> services -> SQLite.
- SQLite First / Offline First: UI and business logic read/write local SQLite first.
- Supabase is remote sync + license/auth source only. Screens must not depend on direct Supabase reads.

## Current Architecture Priorities
- Multi-project app: `project_id` scopes nearly all business data.
- Multi-phase app: `phase_id` scopes invoices, collections, batches/wallets/supplies/reports where applicable.
- License login: first project verification happens through Supabase; after success the device stores project context, not the license number.
- Manager/admin controls active phase. Users inherit current project/phase context automatically.
- Closed phases are read-only. No create/update/delete/approve operations are allowed on closed phases.

## Hard Rules
- Minimal safe diffs only. Do not refactor unrelated code.
- Do not bypass SQLite for business writes.
- Any mutation must: validate -> write SQLite -> enqueue `sync_queue` -> log operation -> notify local subscribers -> sync later.
- Supabase payloads must be schema-safe. Strip local-only fields before remote writes.
- Always respect `project_id`, `phase_id`, active flags, soft-delete rules, and closed-phase guards.
- If `project_id` or required active `phase_id` is missing, block the operation with a clear Arabic error. Do not ask the user to manually enter project id.
- Do not execute Supabase repair SQL until audit + approval. Data integrity comes before UI fixes.
- Never add secrets or service-role keys to the repo or logs.
